Resumen Ejecutivo

El portal proveedor permite administrar el catalogo (categorias y productos) con persistencia en Firestore y soporte de imagenes mediante Firebase Storage. Los productos pasan por un flujo de aprobacion (borrador / pendiente de revision / aprobado / rechazado) para garantizar calidad antes de mostrarse en la app movil.

En este corte, se incluye la edicion de modificadores a nivel de producto como grupos de opciones (obligatorios u opcionales), con items que pueden aportar un costo extra en USD.

M2. Catalogo de productos, categorias y modificadores

Objetivo: Un proveedor puede crear varias categorias, publicar productos asignados a una o mas categorias, y definir modificadores por producto (ej. peso, cantidad, unidad). Los datos se persisten en Firestore y la visibilidad final depende del flujo de revision/aprobacion.

1) Modelo de Datos (Firestore)

Se propone una estructura con subcolecciones por proveedor, aislando datos de cada supplier: suppliers/{providerId}/categories y suppliers/{providerId}/products.

2) Portal Web Proveedor (UI y Flujo)

El modulo de Productos permite alternar entre Categorias y Productos. Las acciones principales se realizan con persistencia directa en Firestore.

Categorias

  • Alta / edicion de categorias (nombre, descripcion opcional e imagen opcional).
  • Activar / desactivar categoria via isActive.
  • Ordenamiento y paginacion basados en updatedAt (ultimo cambio primero).

Productos

  • Alta / edicion de productos con: nombre, descripcion, precio base USD, imagen opcional.
  • Asignacion del producto a una o mas categorias (requerido: minimo 1 categoria).
  • Estado de catalogo: isActive permite activar/desactivar el producto.

Modificadores por Producto

  • Agregar grupos de opciones para modelar variantes o complementos del producto.
  • Configurar grupo como obligatorio u opcional (required).
  • Configurar el modo de seleccion: "Una opcion" (selection=single) o "Varias opciones" (selection=multi).
  • Editar items del grupo: nombre (label) y costo extra en USD (priceDeltaUsd).
  • Desactivar grupos o eliminar items desde la UI (con isActive).

Flujo de Aprobacion: "Enviar a revision"

Antes de ser visible en el marketplace movil, el producto debe pasar por aprobacion del administrador. En el portal proveedor existe un flujo de transicion controlado por estado:

  • draft: borrador (editable).
  • pending_review: el proveedor envia el producto a revision.
  • approved / rejected: estados gestionados por el administrador (segun el modelo de DB propuesto).
  • Boton de UI: "Enviar a revision" (set review.status a pending_review) y "Volver a borrador" (regresa a draft).

Seguridad y visibilidad en marketplace

El modelo de datos y el flujo de aprobacion estan disenados para que los proveedores puedan transicionar un producto solo entre draft y pending_review, mientras que los estados finales (approved, rejected) y la publicacion (publish.isPublished=true) sean controlados por el panel administrador. En la app movil, la lectura se limita a productos activos y aprobados para evitar mostrar contenido no validado.

M3. Promociones

El entregable M3 solicita creacion y gestion de promociones (porcentaje, 2x1, etc.) desde el portal proveedor. En este repositorio, se observa la integracion visual de un panel de "Promociones Activas" en el dashboard, mostrando promociones como lista (con datos de ejemplo).

Importante: en el codigo revisado, la seccion de promociones del dashboard aparece como mock/lista fija y no se evidencia aun el CRUD completo (alta/edicion) desde el portal proveedor, ni el modelo de datos y persistencia especificos para promociones.

Estado actual

  • Dashboard: muestra "Promociones Activas" (visualizacion).
  • Portal proveedor: aun no se confirma el modulo CRUD de promociones con persistencia en Firestore dentro del alcance revisado.

Recomendacion para cierre M3

Para completar M3, se recomienda definir coleccion/modelo de promociones, reglas de seguridad (si aplica), y pantallas CRUD en el portal proveedor, junto con el flujo de aprobacion/publicacion (si el negocio lo requiere).

Mejoras funcionales incluidas (M2)

Adicional a la estructura base, se incorporaron mejoras en la edicion de productos y validaciones para facilitar la carga del catalogo:

  • Editor de descuento con dos modos: USD ($) y porcentaje (%).
  • Calculo de precio final considerando el modo de descuento.
  • Validaciones para evitar descuentos que generen un precio final no valido (cuando aplica).
  • Mejoras de layout y adaptacion a pantallas (grid y secciones responsive).
  • Correccion importante: al editar una categoria, no se fuerza a reactivar automaticamente (se preserva su estado isActive).
