📋 Tienda Go — Portal para proveedores

Este documento resume, qué puede hacer un proveedor en el panel web y cómo se relaciona con lo que ven las tiendas en la aplicación móvil.

Entregas cubiertas en este reporte: M2 (catálogo: categorías, productos y opciones por producto) y M3 (promociones como descuentos por producto).

🎯 ¿Para qué sirve este sistema?

Los proveedores cargan su oferta (productos, precios, fotos y opciones). Las tiendas de conveniencia consultan esa información en la app móvil y pueden generar pedidos.

💡 Idea clave: lo que el proveedor guarda aquí es la “fuente de verdad” de su catálogo. La app solo muestra al público lo que esté aprobado y publicado según las reglas del negocio.

📦 M2 — Catálogo: categorías, productos y “modificadores”

Un proveedor puede organizar su catálogo en categorías (por ejemplo: “Bebidas”, “Snacks”) y crear varios productos. Cada producto puede llevar una o más categorías y, además, grupos de opciones (lo que en el proyecto llamamos modificadores: peso, cantidad, complementos, etc.).

✅ Categorías

  • Nombre y descripción (la descripción es opcional).
  • Foto opcional.
  • Se puede activar o desactivar una categoría sin borrarla (así no se pierde historial).
Ejemplo: categoría “Bebidas frías” con foto de una vitrina. Si la desactivas temporalmente, deja de mostrarse como activa, pero los datos siguen guardados.

🛒 Productos

  • Nombre, descripción, precio en dólares y foto opcional.
  • Debe estar asignado a al menos una categoría.
  • También se puede activar o desactivar el producto.
Ejemplo: “Refresco cola 600 ml” a $1.25, en la categoría “Bebidas frías”, con foto del envase.

⚙️ Modificadores (grupos de opciones)

Cada producto puede tener uno o más grupos. En cada grupo defines un título (por ejemplo: “Cantidad por caja” o “Sabor”), si es obligatorio elegir algo o es opcional, y si el cliente elige una sola opción o varias. Cada opción puede tener un costo extra en dólares.

Ejemplo 1 — obligatorio, una sola opción: “Tamaño” → Pequeño (+$0), Mediano (+$0.50), Grande (+$1.00). El cliente debe elegir uno.

Ejemplo 2 — opcional, varias: “Extras” → Queso (+$0.30), Jalapeños (+$0.20). El cliente puede elegir ninguno, uno o varios.

✉️ Enviar un producto a revisión

Antes de que un producto sea visible para las tiendas en la app, suele pasar por un control de calidad: el proveedor puede pulsar “Enviar a revisión”. Así el equipo administrador ve el producto y puede aprobarlo o pedir cambios.

Ejemplo de flujo: creas un producto en borrador → lo afinas → lo envías a revisión → cuando esté aprobado y publicado según las reglas del negocio, las tiendas lo pueden ver en la app.

🏷️ M3 — Promociones (descuentos en cada producto)

En este proyecto, las promociones no son una pantalla aparte con cupones complicados: están integradas en el editor de cada producto. Puedes aplicar un descuento en dólares ($) o en porcentaje (%). El sistema calcula el precio final que verá la tienda en la app.

Ejemplo A (descuento en dinero): precio lista $10.00, descuento $2.00 → precio final $8.00.

Ejemplo B (descuento en porcentaje): precio lista $10.00, descuento 15% → equivale a $1.50 de rebaja → precio final $8.50.
💡 Para tu equipo: los descuentos se configuran en Productos, al editar el producto. No hace falta un módulo extra de “promociones” en el menú: la oferta va ligada al producto.

🔧 Sección técnica

Esta sección resume decisiones técnicas de diseño y seguridad del módulo de catálogo.

Soft delete (no hard delete)

  • Todas las entidades relevantes operan con soft delete mediante flags (principalmente isActive) y timestamps opcionales (por ejemplo deletedAt).
  • Implicación: el histórico se conserva; los listados operativos filtran por isActive == true.
  • En UI/portal, desactivar equivale a “retirar” del uso sin perder el registro (categorías y productos).

Seguridad / control de acceso

  • El acceso al portal está restringido a usuarios previamente dados de alta como proveedores desde el portal Admin de Tienda Go.
  • Alcance de permisos: un proveedor solo puede leer/escribir dentro de su propio árbol de datos: suppliers/{supplierId}/....
  • El flujo de revisión/publicación está separado por roles:
    • Proveedor: solo transiciona review.status entre draft y pending_review.
    • Admin (panel interno): controla estados finales (approved/rejected) y la publicación (publish.isPublished).

DB design (Firestore) — estructura recomendada

El catálogo se modela con subcolecciones por proveedor para aislar datos y simplificar queries:

  • suppliers/{supplierId}/categories/{categoryId}
  • suppliers/{supplierId}/products/{productId}

Modelo de datos (resumen técnico)

Estructura con subcolecciones por proveedor, aislando datos de cada supplier: suppliers/{providerId}/categories y suppliers/{providerId}/products.

Entidad Campos principales Reglas / notas
Categoría name (requerido), description (opcional), sortIndex (opcional), image (opcional), isActive, createdAt, updatedAt
  • Soporta soft delete con isActive.
  • La UI permite activar/desactivar categorías.
Producto name, description, price (USD), discount (opcional), image (opcional), providerId, categoryIds (1..N), review.status (draft, pending_review, approved, rejected), publish.isPublished, isActive, createdAt, updatedAt
  • Debe pertenecer a al menos una categoría.
  • Visibilidad para marketplace: cuando review.status == approved y publish.isPublished == true (y isActive == true).
  • La UI controla edición y toggles de activación.
Modificador (modelado como grupos de opciones) optionGroups[]:
  • title (ej. “Peso en kg”, “Cantidad por caja”, “Unidad”)
  • required (obligatorio)
  • selection (single o multi)
  • items[]: label, priceDeltaUsd, isActive
  • CRUD: agregar/editar grupos y editar/eliminar ítems (soft delete con isActive).
  • En este corte, los modificadores se representan como opciones (ítems), no como rangos min/max.
Persistencia y archivos: las imágenes de categorías y productos se cargan a Firebase Storage y se guarda el downloadUrl en Firestore. Rutas: supplier-categories/{supplierId}/{categoryId}/image.ext y supplier-products/{supplierId}/{productId}/image.ext. Límite de imagen: 3 MB.

Producto: visibilidad y workflow

  • Relación con categorías: categoryIds: string[] (requerido, longitud ≥ 1).
  • Workflow de revisión: review.status ∈ draft | pending_review | approved | rejected.
  • Publicación: publish.isPublished determina si el producto se expone al marketplace móvil (control admin).
  • Regla de lectura móvil (modelo recomendado): solo se leen productos con isActive == true, review.status == \"approved\" y publish.isPublished == true.

Precio y descuentos (promociones)

  • Moneda: USD.
  • Se almacena los montos en centavos (evita problemas de flotantes): unitAmountCents, amountCents.
  • Descuento soportado por producto vía discount, con tipo amount o percent (según el diseño de esquema).

Opciones / modificadores (Option Groups)

  • Opción recomendada: arrays embebidos en el documento del producto (optionGroups[]) para lecturas eficientes.
  • Cada group define: required, selection, y límites (minSelections/maxSelections), además de items[] con priceDeltaCents.
